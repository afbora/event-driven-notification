## 1. Mission Statement

This project is a technical assessment submission for a Senior Software Engineer (Golang) position. It implements an **Event-Driven Notification System** that ingests notification requests, persists them, queues them for asynchronous processing, dispatches them through multiple channel providers (SMS, Email, Push) with intelligent retry, and exposes both an HTTP API and real-time status updates via WebSocket.

**The reviewers will evaluate:** dockerization, design patterns, unit/feature/integration tests, code style, complexity, frontend skills, error handling, architecture, API design, commit history, this CLAUDE.md file, TDD discipline, and SonarCloud metrics.

**Reading the brief literally is the discipline.** Every technical decision in this project traces back to a sentence in the assessment brief. If a feature, library, or pattern cannot be justified by a sentence in the brief, it does not belong here. This is a deliberate counterweight to over-engineering.

The brief's business context contains four load-bearing phrases that drive every decision:

1. **"millions of notifications daily"** → horizontal scale, stateless services, batch ingestion, connection pooling.
2. **"burst traffic (flash sales, breaking news)"** → asynchronous processing, queue back-pressure, distributed rate limiting.
3. **"retry failed deliveries intelligently"** → exponential backoff with jitter, error classification, circuit breaker, dead letter queue, reconciliation safety net.
4. **"visibility ... for both internal teams and API consumers"** → structured logging, Prometheus metrics, correlation IDs, status query endpoints, notification trace endpoint, and operational alerting.

When in doubt, re-read these four phrases. They are the constitution.

## 2. Interpretation Of The Brief — Decisions Worth Naming

Some interpretation calls shape large parts of the codebase. These are documented openly so reviewers understand the reasoning.

### 2.1 "Failure Handling" Includes Operational Alerting

The brief lists "Failure Handling" as a bonus feature with no further definition. The other bonus items have one-line descriptions; this one does not. We interpret the omission as deliberate — the candidate is expected to define what failure handling means in this context.

Our interpretation: **end-to-end failure response**, comprising:

- **Passive failure handling** (the system recovers on its own): exponential backoff retry with jitter, circuit breaker, dead letter queue, error classification (4xx vs 5xx vs timeout), reconciliation safety net for stuck notifications.
- **Active failure handling** (humans are informed and can act): structured logging, Prometheus alerting rules, AlertManager routing, runbook entries per alert.
- **Observational failure handling** (failure patterns are visible): metrics, dashboards (Grafana), DLQ inspection UI, per-notification trace endpoint.

This interpretation also satisfies the core requirement *"provide visibility into delivery status for both internal teams"* — internal teams do not watch a dashboard 24/7; alerting is how they learn about failures when they happen.

**Consequence:** AlertManager and Grafana are part of `docker compose up`. Alert rules and runbook entries are first-class deliverables, not afterthoughts.

### 2.2 "Real-Time Metrics Endpoint" Means Prometheus

The brief asks for a "real-time metrics endpoint (queue depth, success/failure rates, latency)." We implement this as a Prometheus `/metrics` endpoint with appropriate collectors. This is the de facto standard for backend observability and integrates naturally with the alerting stack from §2.1. A simpler JSON `/api/v1/metrics` endpoint is also exposed for non-Prometheus consumers (a common pattern in production systems where multiple readers exist).

### 2.3 "Structured Logging With Correlation IDs" Means slog + Per-Request ID

Every HTTP request gets a correlation ID at the edge (read from `X-Correlation-ID` header or generated by `CorrelationIDMiddleware`). It flows through the use case, into the queue payload, into the worker, into provider calls, and is present on every log line emitted along the way. One ID, one ULID-shaped string, end-to-end traceable.

**For batch requests, all notifications in the batch share the same correlation ID** — this matches real-world expectations: "this single business action created these N notifications."

### 2.4 "Frontend Skills" Demonstrated Through Operational UIs

We deliberately do not build a custom dashboard. The brief's "frontend skills" criterion is satisfied through a thoughtful stack of operational and developer UIs, each suited to its audience:

- **Swagger UI** (`/docs`) — API consumers
- **asynqmon** — operators monitoring queues, workers, DLQ
- **Grafana** — internal teams viewing business and infrastructure metrics
- **Adminer** — developers/DBAs inspecting the database
- **Redis Commander** — developers inspecting Redis state

This stack mirrors widely-accepted production patterns. Building a bespoke React dashboard would distract from backend quality and duplicate what these tools already provide better. The reasoning is documented in ADR-0005.

### 2.5 "Real-Time Status Updates Via WebSocket"

The bonus item is named literally — "WebSocket Updates" — so we use WebSocket, not SSE. Implementation uses `coder/websocket` (a modern, idiomatic Go WebSocket library). Clients connect to `/api/v1/ws/notifications` and subscribe to specific notification IDs; the server fans out status updates from a Redis pub/sub topic.

### 2.6 Two-Layer Rate Limiting

The brief mentions rate limiting in the context of channel delivery, but production-grade APIs need **two independent rate limiters** doing different jobs:

- **Inbound:** protects the API from abuse — 60 requests/minute per client IP, applied at the HTTP middleware layer.
- **Outbound:** protects external providers from overload — 100 messages/second per channel, applied at the worker layer before each provider call.

These are completely separate concerns and use separate Redis keys. Conflating them is a common bug.

### 2.7 No `.env` File — All Config In `docker-compose.yml`

The system is designed for one-command bring-up: `docker compose up`. Every required environment variable lives in `docker-compose.yml` with a working default. This means:

- A reviewer can clone the repo and reach a working API in under 60 seconds.
- There is no `.env.example` to copy, no secrets to set, no chicken-and-egg.
- Production overrides happen at the orchestration layer (Kubernetes ConfigMap, secrets manager) — not via a local file format.

This is documented in ADR-0010 with the trade-offs spelled out.

## 3. Non-Negotiable Project Principles

These are the rules. Violating any of them is a defect, regardless of how clever the violating code is.

### 3.1 Feature Scope Discipline

- **Implement only what the brief requires or what bonus items the plan calls out.** Do not add features because they would be "cool" or "professional." The reviewers will not reward extra features; they will reward a small surface area executed flawlessly.
- If you have an idea for a feature not in `PLAN.md`, write it down in a `// FUTURE:` comment and move on. Do not implement it.
- Exception: features added under a documented interpretation in §2 are in scope. The interpretation itself is what we are defending in a review.

### 3.2 Quality Without Compromise

- **Code quality, architecture, performance, and observability are unbounded.** Every commit must leave the codebase in a state you would be proud to ship to production.
- Tests are not optional. No commit introduces production code without a corresponding test commit immediately preceding it (TDD). See §7.
- Lint must pass. Coverage must not decrease. CI must be green before any PR is merged.

### 3.3 Domain Purity

- `internal/domain/` and `internal/application/` import **only** the Go standard library and other packages within those two directories. No `database/sql`, no `net/http`, no `asynq`, no `redis`. Ever.
- If application logic needs to interact with the outside world, it does so through a **port** (interface) defined in `internal/ports/`. Adapters in `internal/adapters/` implement those ports.
- This is the **Dependency Inversion Principle** made physical. The reviewer should be able to delete `internal/adapters/postgres/` and the domain still compiles.

### 3.4 SOLID, Applied Concretely

- **S** — Each handler does one thing. Each use case does one thing. Repositories do not contain business logic.
- **O** — New channels (e.g., WhatsApp) are added by writing a new `Provider` implementation. No `switch` statements on channel type in business logic.
- **L** — Mock providers are substitutable for real providers in every test.
- **I** — Ports are narrow. `NotificationReader` and `NotificationWriter` are separate if a consumer needs only one.
- **D** — Use cases depend on `ports.Queue`, never `*asynq.Client`.

### 3.5 Errors Are Values, Not Strings

- Define sentinel errors and error types in `internal/domain/errors.go` (e.g., `ErrNotificationNotFound`, `ErrInvalidChannel`).
- Adapter errors are wrapped with `fmt.Errorf("postgres: scan notification: %w", err)`.
- HTTP handlers translate domain errors to RFC 7807 Problem Details responses via a single translation function. No `if err.Error() == "not found"` comparisons. Use `errors.Is` and `errors.As`.
- Provider errors are classified into **permanent** (4xx → no retry) and **transient** (5xx, timeout, network error → retry with backoff). The `DeliveryResult` DTO carries this classification.
- Never `panic` in production code paths. `panic` is reserved for "this is a programmer bug, the process must die" situations during startup.

### 3.6 Context Everywhere

- Every function that performs I/O, queues work, or could be cancelled takes `ctx context.Context` as its first parameter.
- Never store contexts in structs. Never pass `context.TODO()` outside of tests and `main()`.
- Respect cancellation: long loops must `select` on `ctx.Done()`.

### 3.7 Configuration Is External — But Without `.env`

- All configuration comes from environment variables (12-Factor).
- `docker-compose.yml` is the single place that defines them for local/CI use.
- A single `internal/infrastructure/config` package loads and validates config at startup. If config is invalid, the process refuses to start with a clear error.

### 3.8 Observability Is A Three-Layer Stack

- **Layer 1 — Telemetry:** Logs, metrics, traces emitted by the application. Logs via `log/slog`, metrics via Prometheus client, traces via OpenTelemetry SDK (no-op exporter by default).
- **Layer 2 — Collection:** Prometheus scrapes `/metrics`. AlertManager receives alerts from Prometheus. Grafana queries Prometheus for dashboards.
- **Layer 3 — Response:** Alert rules in `deploy/prometheus/alerts.yml`. Runbook entries in `docs/RUNBOOK.md` for every alert. AlertManager configured to log received alerts (production would route to Slack/PagerDuty).

All three layers are part of `docker compose up`. The system is operationally complete out of the box.

### 3.9 Idempotency Is Two-Layered

- **API layer:** `Idempotency-Key` request header. Redis stores the response for 24 hours. Duplicate requests return the cached response (status 200, not 201/202) without re-processing.
- **Worker layer:** A unique constraint on `notifications.idempotency_key` in PostgreSQL guarantees that even if the queue somehow delivers a duplicate task, the worker will not double-send.

### 3.10 Atomic Status Claim Prevents Double Processing

When the worker dequeues a task, it must atomically claim the notification before processing:

```sql
UPDATE notifications
SET status = 'processing', updated_at = NOW()
WHERE id = $1 AND status IN ('queued', 'retrying')
RETURNING *;
```

If `RowsAffected = 0`, another worker (or a re-delivery from asynq under network partition) already claimed it. The worker logs and exits cleanly without calling the provider.

This belt-and-braces approach handles edge cases where:
- A network partition causes asynq to redeliver a task it believes was lost.
- A horizontally scaled worker pool has multiple workers consuming the same queue.
- A reconciliation safety net (see §3.11) re-enqueues a notification that is actually still being processed.

### 3.11 Reconciliation Safety Net For Stuck Notifications

Even with perfect retry logic, notifications can get stuck:
- **Worker crashes mid-processing** → notification stuck in `processing` state.
- **Redis flushes** → in-flight retry schedule is lost.
- **Database deadlock** → status update fails silently.
- **API-to-queue write fails after DB commit** → notification orphaned in `pending` state, never enqueued. (This is the dual-write race documented in ADR-0011.)

A reconciliation job runs every minute (via the `cmd/reconciler` binary) and:

1. Finds notifications in `processing` for more than 5 minutes → marks `failed` with reason `worker_timeout`.
2. Finds notifications in `retrying` with `next_retry_at < NOW() - 1 minute` → re-enqueues them.
3. Finds notifications in `pending` for more than 5 minutes → re-enqueues them. (This catches the dual-write race: notification persisted but enqueue failed.)

**Concurrency:** The reconciler uses PostgreSQL's `SELECT ... FOR UPDATE SKIP LOCKED` semantics when claiming rows for reconciliation. This allows running multiple reconciler instances horizontally without conflicting claims — each instance locks only the rows it can process; competing instances skip locked rows without waiting. If a reconciler crashes mid-transaction, the lock is released automatically and the row becomes visible again to other reconcilers.

This is the "safety net under the safety net." It exists because production systems eventually face conditions retry logic alone cannot recover from.

### 3.12 Documentation Co-Exists With Code

- Every exported identifier (type, function, constant) in `internal/domain/` and `internal/application/` has a Go doc comment that explains **what it is and what invariants it preserves**, not just what its name says.
- Architectural decisions live in `docs/adr/NNNN-title.md` (one file per decision). A new ADR is written whenever a non-trivial choice is made.
- Every endpoint in `api/openapi.yaml` has a clear description, parameter docs, response schema, and at least one example.

## 4. Anti-Patterns — Things That Must Not Happen

These are concrete failure modes seen in real codebases. Each one is a hard "no" in this repo.

- ❌ **ORM in the data layer.** No GORM, no Ent. Use `sqlc` to generate type-safe Go from raw SQL. Migrations are SQL files managed by `golang-migrate`.
- ❌ **Business logic in HTTP handlers.** Handlers parse input, call a use case, translate the result. That's it.
- ❌ **Returning `interface{}` or `any` from domain functions.** Use concrete types or generics.
- ❌ **`init()` functions that do more than register.** Initialization happens in `main()` where dependencies are explicit.
- ❌ **Global variables that hold state.** Pass dependencies explicitly. The only acceptable globals are the Prometheus registry and the slog default logger, both set in `main()`.
- ❌ **String comparison on error messages.** Use `errors.Is` and `errors.As`.
- ❌ **`time.Now()` called directly inside business logic.** Inject a `ports.Clock` so tests can control time.
- ❌ **Goroutines without `context.Context` and without a wait mechanism.** Every goroutine is owned, cancellable, and joinable.
- ❌ **Empty error handling** (ignoring errors with `_`). Either handle the error, wrap and return it, or document why ignoring is correct.
- ❌ **Tests that share state.** Each test is independent and can run in any order. Use `t.Cleanup` for teardown.
- ❌ **Mocking what you don't own.** Wrap third-party clients in a port, mock the port. Do not mock `*asynq.Client` directly.
- ❌ **`fmt.Println` for debugging left in code.** Use `slog.Debug` if a log is intentional; otherwise delete.
- ❌ **Untyped JSON in API responses.** Every response has a Go struct with `json:` tags and is documented in OpenAPI.
- ❌ **CORS wildcard `*` in production paths.** The example config locks CORS to known origins (asynqmon, Grafana).
- ❌ **Hardcoded sleeps in tests.** Use `assert.Eventually` or proper synchronization. Sleeps make CI flaky.
- ❌ **Alerts without runbooks.** Every alert rule in `alerts.yml` has a corresponding entry in `RUNBOOK.md` explaining what triggered it and what an operator should do.
- ❌ **Skipping the atomic claim in the worker.** Always `UPDATE ... WHERE status IN (...)` and check `RowsAffected` before proceeding.

## 5. Architecture Overview

The system follows **Hexagonal Architecture (Ports & Adapters)**. Three concentric layers:

```
┌─────────────────────────────────────────────────────────────┐
│ Adapters (internal/adapters/)                               │
│ http, postgres, redis, asynq, provider, websocket           │
│ Knows about: frameworks, drivers, protocols                 │
└──────────────────────┬──────────────────────────────────────┘
                       │ implements
                       ▼
┌─────────────────────────────────────────────────────────────┐
│ Ports (internal/ports/)                                     │
│ Repository, Queue, Provider, IdempotencyStore, RateLimiter, │
│ StatusBroadcaster, Clock                                    │
│ Pure interfaces, no implementation                          │
└──────────────────────┬──────────────────────────────────────┘
                       │ used by
                       ▼
┌─────────────────────────────────────────────────────────────┐
│ Application (internal/application/)                         │
│ Use cases: CreateNotification, ProcessNotification, etc.    │
│ Orchestrates domain objects via ports                       │
└──────────────────────┬──────────────────────────────────────┘
                       │ uses
                       ▼
┌─────────────────────────────────────────────────────────────┐
│ Domain (internal/domain/)                                   │
│ Entities, value objects, domain errors, invariants          │
│ Pure Go, zero dependencies outside stdlib                   │
└─────────────────────────────────────────────────────────────┘
```

**Three binaries, one image.** `cmd/api`, `cmd/worker`, `cmd/reconciler` are separate `main` packages but share `internal/`. The same Docker image runs all three; the entrypoint chooses which `main` to invoke based on the container's command. This mirrors how production deployments scale each tier independently.

**Data flow (happy path):**

1. Client `POST /api/v1/notifications` → API handler validates → use case persists notification (`status=pending`) with `idempotency_key` (if provided) → use case enqueues asynq task → API returns `202 Accepted` with notification ID and `Location` header. A `notification_logs` entry is written: event `created`.
2. Worker dequeues task → **atomically claims** notification (`UPDATE ... WHERE status IN ('queued', 'retrying')`) → checks inbound channel rate limit → resolves provider via strategy registry → calls provider behind circuit breaker → updates notification (`status=delivered|failed|retrying`) → logs attempt to `notification_logs` → publishes status update to Redis pub/sub. WebSocket hub receives the update and fans out to subscribed clients.
3. WebSocket client `GET /api/v1/ws/notifications` upgrades the connection → subscribes by sending `{"action": "subscribe", "notification_id": "..."}` → receives JSON status updates as the notification progresses.

**Failure paths:**

- Provider returns 5xx or times out → marked `retrying` → asynq retries with exponential backoff + jitter (configurable: `30s * 2^(attempt-1) + jitter(0, 30s)`) → after `max_attempts` (default 5), task moves to dead letter queue and `notification.status=failed`.
- Provider returns 4xx → no retry (permanent failure) → `notification.status=failed` immediately with reason from response.
- Circuit breaker opens after 5 failures in 10 seconds → subsequent attempts fail fast for 30 seconds → half-open probe → close on success.
- Redis unreachable → API returns 503 (readiness probe also fails); worker pauses processing until Redis recovers.
- Worker crashes mid-processing → notification remains in `processing` → reconciler picks it up after 5 minutes and marks `failed` with reason `worker_timeout`.

**Observability paths:**

- Every notification's lifecycle is recorded in `notification_logs` table → exposed via `GET /api/v1/notifications/{id}/trace` for support and debugging.
- Every metric crosses a threshold → Prometheus evaluates alert rule → AlertManager receives → routes to log receiver (configurable to Slack/PagerDuty/email in production).
- Operator opens Grafana → sees pre-built dashboards for queue depth, throughput, success rate per channel, circuit breaker state.
- Operator opens runbook entry for the firing alert → follows step-by-step remediation.

## 7. Test-Driven Development — How We Work

TDD here is not aspirational; it is the workflow. Reviewers will read the commit log and see the rhythm.

**The cycle for every unit of work:**

1. Write a failing test that describes the desired behaviour.
2. Commit: `test(scope): describe what fails`
3. Write the minimum production code to make the test pass.
4. Commit: `feat(scope): implement what passes`
5. Refactor if needed; tests stay green.
6. Commit: `refactor(scope): describe the cleanup`

**Test categories and where they live:**

- **Unit tests** (`*_test.go` next to the code) — domain and application logic. Pure, fast, no I/O. Coverage target: 90%+.
- **Integration tests** (`tests/integration/`) — adapters against real Postgres and Redis spun up via `testcontainers-go`. Tagged with `//go:build integration` so they run separately. Coverage target: 70%+ for adapter code.
- **End-to-end tests** (`tests/e2e/`) — HTTP request through the full stack to a `MockProvider`. Validates contracts, status transitions, retry behaviour, WebSocket fan-out. Tagged `//go:build e2e`.
- **Load tests** (`tests/load/`) — k6 scripts validating throughput, burst absorption, and rate-limit behaviour. Not part of `make test`; run separately via `make load-test`. Results documented in `docs/LOAD_TEST.md`.

**What we test:**

- Happy paths AND failure paths. Every error branch has a test.
- State transitions (e.g., `pending → queued → processing → delivered` is allowed; `delivered → anything` is not).
- Concurrency: rate limiter under contention, idempotent enqueue from parallel callers, atomic claim under simulated worker race.
- Timeouts and cancellation: context cancellation aborts in-flight work cleanly.
- WebSocket: connection upgrade, subscribe/unsubscribe, fan-out, disconnect cleanup.

**What we do NOT test:**

- Third-party libraries (we test our adapter, not asynq's internals).
- Framework boilerplate (chi routing, middleware wiring).
- Trivial getters and setters.

**Table-driven tests are the default** for any function with more than one branch. Subtests use `t.Run(tc.name, ...)`.

## 8. Commit & Branch Conventions

### Branching

- `main` is always green and deployable.
- Feature branches: `feat/short-description`, `fix/short-description`, `chore/short-description`, `docs/short-description`.
- One PR per logical change. PRs are self-reviewed and merged via merge-commit **only if** the branch's commit history is clean.
- Use **merge commits** (not squash) so the TDD rhythm remains visible in `main`'s history.

### Commit Messages — Conventional Commits

Format: `<type>(<scope>): <subject>` — imperative mood, lowercase, no trailing period.

**Types:** `feat`, `fix`, `test`, `refactor`, `chore`, `docs`, `perf`, `ci`, `build`.

**Examples (these are the kind of messages we want to see):**

```
test(domain): add failing test for notification status transitions
feat(domain): implement notification status state machine
test(application): add failing test for create notification use case
feat(application): implement create notification use case
test(adapter/postgres): add failing test for atomic status claim
feat(adapter/postgres): implement atomic status claim in worker repository
refactor(application): extract idempotency check into dedicated method
docs(adr): add ADR-0007 on circuit breaker thresholds
ci: add SonarCloud upload step to backend workflow
```

**Bad commits (do not produce these):**

```
update stuff                 ← no scope, no detail
WIP                          ← never
fix bug                      ← which bug?
feat: add everything         ← too broad
Fixed the failing test       ← past tense, capitalized, vague
```

### Commit Execution

**The human runs all `git` commands.** Claude Code prepares the change set, then surfaces:

1. The list of files changed (path + one-line summary).
2. The exact commit message.
3. The exact `git` command sequence (`git add ...`, `git commit -m "..."`, optional `git push`).

The human reviews, runs the commands, and confirms. Claude Code does **not** run `git commit` or `git push` itself in this project.

## 9. Coding Style — Go

- `gofmt` and `goimports` enforced via `golangci-lint`. No manual formatting debates.
- Receiver names: short (one or two letters), consistent across all methods of a type.
- Acronyms: ALL CAPS in identifiers (`HTTPClient`, not `HttpClient`; `ID`, not `Id`).
- Errors are returned, not logged-and-returned. Log at the boundary (handler or worker entry point), once.
- Avoid `else` after `return`. Use early returns.
- Group imports: stdlib, third-party, internal — separated by blank lines, `goimports` will sort within groups.
- Comments on exported identifiers start with the identifier name: `// Notification represents...`.
- Tests use `require` for "stop on failure" assertions, `assert` for "report and continue" assertions. Default to `require`.

## 10. API Design Conventions

- Versioned base path: `/api/v1/...`.
- Resources are plural nouns: `/notifications`, `/templates`.
- HTTP verbs map to actions: `POST` create, `GET` read, `PATCH` partial update (e.g., cancel), `DELETE` removal (rarely used here).
- **Cancellation is `PATCH /api/v1/notifications/{id}/cancel`** — it changes state, not removes the resource.
- **Pagination is cursor-based**, not offset-based. Response includes `next_cursor` and `prev_cursor`. Rationale: offsets become incorrect under concurrent writes at scale.
- Filters are query parameters: `?status=delivered&channel=sms&created_after=2026-01-01T00:00:00Z`.
- Errors are RFC 7807 Problem Details JSON:
  ```json
  {
    "type": "https://example.com/probs/invalid-channel",
    "title": "Invalid channel",
    "status": 400,
    "detail": "Channel must be one of: sms, email, push",
    "instance": "/api/v1/notifications",
    "correlation_id": "01HXYZ..."
  }
  ```
- Idempotency: `Idempotency-Key` request header (UUID v4 or ULID). Cached response (status `200`) returned on repeat within 24 hours; first request returns `201`/`202`.
- Correlation ID: server reads `X-Correlation-ID` if present, otherwise generates a ULID. Always echoed in response header and embedded in every log line and queue payload. Batch requests share the same ID across all notifications in the batch.
- **Two-layer rate limiting:** inbound `60 req/min/IP` (returns `429`); outbound `100 msg/sec/channel` (releases the worker task back to the queue with 1s delay).
- **Notification trace:** `GET /api/v1/notifications/{id}/trace` returns the ordered status transition log for support/debugging.
- **WebSocket:** `GET /api/v1/ws/notifications` upgrades to WebSocket. Client subscribes per notification ID; server fans out status updates from Redis pub/sub.
- OpenAPI spec is the source of truth. Code is generated from it; documentation is served from it at `/docs`.

## 11. Database Conventions

- Table names are plural snake_case: `notifications`, `notification_logs`, `templates`.
- Primary keys are `id UUID` — UUID v7 generated by the application (lexicographically sortable, avoids InnoDB-style index fragmentation in PG too).
- Timestamps: `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`. The application is the source of truth for `updated_at` — every mark method on a domain entity sets it from an injected clock, and repositories include it in their UPDATE payloads. A trigger-based fallback was tried and dropped in migration 000002; it conflicted with deterministic timestamps in tests.
- Soft delete is **not** used. Cancellations set `status='cancelled'`. Hard deletes happen only for templates and only when no notifications reference them.
- Indexes: every column used in a `WHERE` clause that is not the PK gets an index. Composite indexes for common filter combinations.
- **Required indexes for performance:**
  - `notifications(status, scheduled_at)` — for reconciler scans
  - `notifications(idempotency_key)` UNIQUE — for idempotency
  - `notifications(batch_id)` — for batch status queries
  - `notification_logs(notification_id, created_at)` — for trace endpoint
- Migrations: numbered `NNNNNN_description.up.sql` and `NNNNNN_description.down.sql`. Every `up` has a corresponding `down`. Migrations are immutable once committed to `main`.
- No raw string concatenation for queries. `sqlc` generates parameterised queries.

## 12. Observability Conventions

### 12.1 Metrics

Metric names follow Prometheus conventions: `<namespace>_<subsystem>_<name>_<unit>`. Our namespace is `notifications`.

Required metrics (defined in `internal/infrastructure/metrics`):

- **Counters:** `notifications_created_total{channel,priority}`, `notifications_delivered_total{channel}`, `notifications_failed_total{channel,reason}`, `notifications_attempts_total{channel,outcome}`, `http_requests_total{method,path,status}`, `inbound_rate_limit_hits_total{endpoint}`, `outbound_rate_limit_hits_total{channel}`.
- **Gauges:** `notifications_queue_depth{queue}`, `notifications_circuit_breaker_state{provider}` (0=closed, 1=open, 2=half-open), `notifications_websocket_clients{}`.
- **Histograms:** `notifications_processing_duration_seconds{channel}` (buckets: 0.01, 0.05, 0.1, 0.5, 1, 5, 10, 30), `http_request_duration_seconds{method,path}` (buckets: 0.005, 0.01, 0.05, 0.1, 0.5, 1, 5).

### 12.2 Logging

Every log line is JSON with these fields, always:

- `time` (RFC3339 with nanoseconds)
- `level` (DEBUG, INFO, WARN, ERROR)
- `msg` (human-readable summary)
- `service` (api or worker or reconciler)
- `correlation_id` (ULID, propagated from request)

Plus context-specific fields: `notification_id`, `channel`, `provider`, `attempt`, `duration_ms`, etc.

### 12.3 Notification Trace

Every status transition writes a row to `notification_logs`:

| Column | Type | Notes |
|---|---|---|
| `id` | UUID | PK |
| `notification_id` | UUID | FK to notifications |
| `correlation_id` | TEXT | From request |
| `event` | TEXT | `created`, `queued`, `processing`, `delivered`, `retrying`, `failed`, `cancelled` |
| `details` | JSONB | Optional context (provider response, error message, retry attempt) |
| `created_at` | TIMESTAMPTZ | Event time |

Exposed via `GET /api/v1/notifications/{id}/trace` for support and debugging.

### 12.4 Tracing

OpenTelemetry SDK is wired up. The default exporter is no-op (no Jaeger container). Each HTTP request, queue enqueue, queue dequeue, and provider call creates a span. To enable real tracing, set `OTEL_EXPORTER_OTLP_ENDPOINT` and add a tracing backend to compose; nothing else changes.

### 12.5 Alerting (See §3.8)

Every alert rule in `deploy/prometheus/alerts.yml` has:

- A clear `alert:` name (PascalCase).
- A `for:` duration (do not page on momentary blips).
- A `severity:` label (`critical`, `warning`, `info`).
- An `annotations.summary` (one line).
- An `annotations.runbook_url` pointing to the corresponding section in `docs/RUNBOOK.md`.

## 13. How Claude Code Should Behave In This Repo

This section is the most important one for you, Claude Code. Read it twice.

### 13.1 Before You Write Any Code

1. **Read `PLAN.md` and locate the current phase.** Do not work ahead.
2. **Read the brief** in `docs/brief.pdf` if you need context on a requirement.
3. **Check `.claude/skills/`** — there are 15 skill files organised by category (core, quality, operations). When your task matches a skill's purpose, follow that skill's recipe. Do not invent a different approach.
4. **Read existing code in the area you're modifying.** Match its style.
5. **Run `make test` and `make lint`** to confirm a green baseline before changing anything.

### 13.2 When You Are Working

- Stay within the current phase. If a task naturally requires touching the next phase, **stop and ask**.
- Write the test first. Always. Even when the implementation is "obvious."
- When you encounter an ambiguity not resolved by this document, `PLAN.md`, or the relevant skill, **ask the human**. Do not guess.
- When you make a non-trivial decision (a library choice, an architectural shape, a performance trade-off), **write an ADR** in `docs/adr/` before writing the code. Use the `add-adr` skill.
- Keep changes small. One concept per commit. If you find yourself wanting to commit "and also...", split it.

### 13.3 When You Are Done With A Task

**First, verify locally.** Before surfacing any commit, run the appropriate check set and include the result (pass/fail summary) in the surface so the human sees the green signal:

- For `feat` / `refactor` / `fix` commits — run the full triplet, all three must succeed before surfacing:
  ```bash
  go build ./...
  go test ./...
  golangci-lint run ./...
  ```
  If any check fails, fix it first; do not push the breakage onto the human.

- For `test` commits in the TDD red phase — run `go test ./...` only and explicitly note that the build is expected to fail (the implementation hasn't landed yet). Skip `golangci-lint` because the package will not compile.

- For `docs` / `chore` / `ci` commits — run whichever subset is meaningful (e.g. `docker compose config --quiet` for a compose change, the `make help` listing for a Makefile change). Use judgement; the goal is "no surprises after the human commits."

**Then surface to the human:**

1. **What changed:** files added, modified, deleted, with a one-line reason each.
2. **What was tested:** which tests were added, which existing tests still pass.
3. **Commit message(s)**, exactly as they should appear in git.
4. **Commit command sequence:**
   ```bash
   git add <files>
   git commit -m "<message>"
   ```
5. **Anything the human should review carefully** before running the commands.

Then wait. Do not move on until the human confirms.

### 13.4 What You Must Not Do

- Do not run `git commit`, `git push`, `git checkout`, or any state-mutating git command. Surface the commands; the human runs them.
- Do not install new dependencies without asking. The stack in §6 is fixed.
- Do not modify `CLAUDE.md`, `PLAN.md`, or any skill in `.claude/skills/` without an explicit instruction.
- Do not add features outside the current phase.
- Do not refactor "while you're in there." Refactors are their own commits and their own decisions.
- Do not skip writing tests because "this is trivial." Trivial code is where bugs hide.
- Do not introduce a new pattern when an existing one in this codebase already solves the problem.
- Do not silence linter warnings with `//nolint` unless you also add a comment explaining why and a `// TODO:` to fix it properly.
- Do not create a `.env` file. All config lives in `docker-compose.yml` (see §2.7).

### 13.5 When You Disagree

You will sometimes have a strong technical opinion that differs from what this document says. **Raise it.** Explain your reasoning. The human will either update the document or explain why the original decision stands. Either way, the disagreement gets resolved before code changes.

What you must not do: silently do it your way and hope no one notices. The reviewer will notice.

### 13.6 Working With Skills

The `.claude/skills/` directory contains 15 skill files organised into three categories:

- **`core/`** (4 skills) — Day-to-day development tasks: `add-use-case`, `add-endpoint`, `add-migration`, `add-provider`.
- **`quality/`** (7 skills) — Testing and code health: `add-integration-test`, `add-e2e-test`, `debug-failing-test`, `refactor-without-breaking`, `error-handling-review`, `check-hexagonal-boundaries`, `add-prometheus-metric`.
- **`operations/`** (4 skills) — Cross-cutting and operational concerns: `add-alert-rule`, `update-openapi`, `add-adr`, `add-ci-job`.

**When to consult a skill:** Whenever your task matches one of these names. Even if you "know how" to do it, the skill encodes this project's specific conventions.

**When to skip a skill:** When no skill matches the task. Most one-off chores (configuration tweaks, doc fixes, dependency updates) do not have a skill.

**If a skill is wrong:** Tell the human. Do not modify it on your own. The skill is updated, then you follow the updated version.

## 14. Definition of Done

A unit of work is **done** when all of the following are true:

- [ ] Code is written and committed.
- [ ] Tests are written and passing (unit + integration where applicable).
- [ ] `make lint` passes with zero warnings.
- [ ] `make test` passes with no skipped tests (other than build-tag-gated ones).
- [ ] Coverage for changed packages has not decreased.
- [ ] OpenAPI spec is updated if any HTTP contract changed.
- [ ] Documentation (Go doc comments, README sections, ADR if applicable) is updated.
- [ ] If the change introduces a new failure mode, an alert rule and runbook entry exist for it.
- [ ] The change is described in a commit message that a stranger could understand.
- [ ] PR description summarises the change and references the phase in `PLAN.md`.

A phase in `PLAN.md` is **done** when all its tasks meet the above, the CI pipeline is green, and the human has merged the PR(s).
